# Дизайн: переносимый модуль яркости для Waybar в Omarchy

## Цель

Добавить в Waybar модуль яркости с иконкой солнца и отображением текущего уровня яркости в процентах.

Требования к пользовательскому поведению:

- модуль показывает иконку солнца и текущую яркость в `%`
- яркость меняется колесом мыши по модулю
- модуль располагается в `modules-right` перед `battery`
- если ни один дисплей не поддерживает управление яркостью, модуль не исчезает, а показывает только серую иконку солнца
- решение должно переноситься на другое устройство с Omarchy без ручной переработки структуры

## Почему отдельный каталог фичи

Репозиторий уже устроен как набор самостоятельных Omarchy-фич. Поэтому модуль яркости должен жить как отдельный каталог проекта, а не как набор разрозненных локальных правок в `~/.config/waybar/`.

Это дает:

- предсказуемую структуру установки
- воспроизводимость на другом устройстве
- изоляцию логики и файлов от остального конфига пользователя
- возможность удаления через отдельный `uninstall.sh`

## Выбранная архитектура

Фича оформляется как отдельный проект, например:

```text
omarchy-waybar-brightness/
├── README.md
├── install.sh
├── uninstall.sh
├── waybar/
│   └── brightness/
│       ├── brightness-lib.sh
│       ├── brightness-status.sh
│       ├── brightness-control.sh
│       └── brightness.css
```

Внутри логика разделяется на три слоя:

1. `detect`
Определяет доступные устройства яркости:
- встроенный экран через `backlight`
- внешние мониторы через DDC/CI

2. `status`
Получает текущее состояние выбранного устройства и отдает JSON для Waybar.

3. `change`
Меняет яркость по направлению `up` или `down`.

## Интеграция в Omarchy и Waybar

Фича не должна менять Omarchy-исходники в `~/.local/share/omarchy/`.

Разрешенные точки интеграции:

- файлы пользователя в `~/.config/waybar/`
- перезапуск через `omarchy-restart-waybar`

Сценарий установки:

1. Скопировать модульные файлы в `~/.config/waybar/brightness/`
2. Добавить модуль `custom/brightness` в `~/.config/waybar/config.jsonc`
3. Вставить модуль в `modules-right` перед `battery`
4. Подключить стили модуля в `~/.config/waybar/style.css`
5. Выполнить `omarchy-restart-waybar`

Сценарий удаления:

1. Удалить модуль из `config.jsonc`
2. Удалить подключенный CSS-блок или `@import`
3. Удалить `~/.config/waybar/brightness/`
4. Выполнить `omarchy-restart-waybar`

## Поведение модуля

Waybar-модуль должен быть объявлен как `custom/brightness`.

Ожидаемое поведение:

- `exec` вызывает `brightness-status.sh`
- `return-type` установлен в `json`
- `on-scroll-up` увеличивает яркость
- `on-scroll-down` уменьшает яркость
- `interval` составляет 2-5 секунд

Отображение:

- при доступном устройстве: активная иконка солнца и текст вида `73%`
- при недоступности: только иконка солнца без процентов, в сером состоянии

Клик по модулю не используется. Функция управления ограничивается отображением состояния и регулировкой колесом мыши.

## Правило выбора устройства

Чтобы избежать лишнего UI и при этом сохранить переносимость, устройство выбирается автоматически по фиксированному приоритету:

1. если найден встроенный `backlight`, использовать его
2. если встроенного экрана нет, но найден один DDC/CI-монитор, использовать его
3. если найдено несколько DDC/CI-мониторов, использовать первый в стабильной сортировке

В первой версии не добавляется переключатель устройств, меню выбора монитора или popup-слайдер. Эти расширения возможны позже без изменения базовой структуры проекта.

## Технический подход

### `brightness-lib.sh`

Общая библиотека должна инкапсулировать:

- поиск доступных устройств
- определение типа устройства: `backlight` или `ddc`
- чтение текущего процента
- изменение яркости на шаг
- единый формат возврата данных для status/control-скриптов

Логика не должна быть привязана к конкретной модели устройства, хардкодным путям подсветки или заранее заданным monitor id.

### `brightness-status.sh`

Скрипт читает состояние через библиотеку и печатает JSON для Waybar.

Минимально он должен отдавать:

- `text` для видимого содержимого модуля
- `class` для активного или неактивного состояния
- `tooltip` с краткой диагностикой, если это уместно

### `brightness-control.sh`

Скрипт принимает аргумент `up` или `down`, выбирает то же активное устройство и меняет яркость на фиксированный шаг.

Шаг изменения должен быть одинаковым для обеих веток (`backlight` и `ddc`), чтобы поведение было предсказуемым для пользователя.

## Зависимости и деградация

Поддержка должна быть построена по возможностям системы:

- для встроенной подсветки использовать `brightnessctl`, если доступен и видит устройство
- для внешних мониторов использовать `ddcutil`, если доступен и монитор поддерживает DDC/CI

Если одна из веток недоступна, это не должно ломать фичу целиком.

Допустимое поведение:

- есть `backlight` -> модуль работает через него
- нет `backlight`, но есть рабочий `ddcutil` -> модуль работает через DDC/CI
- нет ни одного доступного способа -> модуль остается в bar как серая иконка солнца

## Установка на другое устройство

Фича должна переноситься как самостоятельный каталог проекта.

Ожидаемый пользовательский поток:

1. клонировать или скопировать каталог `omarchy-waybar-brightness/`
2. запустить `install.sh`
3. при необходимости установить системные зависимости, если их не хватает
4. получить рабочий модуль в Waybar без ручной правки нескольких несвязанных файлов

Install-скрипт должен быть идемпотентным:

- повторный запуск не должен дублировать блоки в `config.jsonc`
- повторный запуск не должен повторно вставлять CSS-подключение
- повторный запуск должен обновлять уже установленные файлы модуля

## Диагностика и документация

`README.md` должен объяснять:

- как установить фичу
- какие зависимости нужны
- как проверить, какие устройства яркости обнаружены
- как понять, почему модуль стал серым
- как удалить фичу

Минимальная диагностика должна покрывать:

- отсутствие `brightnessctl`
- отсутствие `ddcutil`
- отсутствие доступных устройств
- ситуацию, когда монитор физически подключен, но DDC/CI недоступен

## Тестирование

Проверка реализации должна включать:

- ноутбук со встроенным экраном
- систему только с внешним DDC/CI-монитором
- систему без доступного управления яркостью
- повторный запуск `install.sh`
- `uninstall.sh` после установки

На уровне пользовательского результата нужно подтвердить:

- корректное размещение модуля перед `battery`
- изменение яркости по скроллу вверх и вниз
- отображение процентов при доступном управлении
- серое состояние без процентов при недоступности

## Что не входит в первую версию

В первую версию намеренно не входят:

- popup со слайдером яркости
- выбор конкретного монитора по клику
- циклическое переключение между несколькими мониторами
- хранение пользовательских профилей яркости

Это отложено, чтобы сохранить минимальную и стабильную архитектуру для реального использования в Omarchy.
